SQL Databases

  • Data is stored in database tables by following a strict data schema (= structure)
  • Data is distributed across multiple tables which are connected via relations
SQL stands for Structured Query Language and it therefore is not a database itself but only a query language you can use to interact with a specific type of database. SQL allows you to store, update, delete and of course retrieve data from relational database management systems (RDBMS). You can't add records which don't adhere to this schema. Got more fields? Sorry, you have to pick another table. Missing some field data? Not your table then!

The Problem with Structured Data

The problem with a structured database is that it can become too structured and require a lot of resources to create and design a good database. Just to show one example of this, what if we have a table that defines some structure and for a specific row we want to have some extra data? This is not easily done with a relational database.

NoSQL Databases

  • No schemas
  • No relations
There is a core difference: You can put data of different structure into the same collection. You couldn't do that in the SQL world - each table had a clear schema there.

Types of noSQL databases

There are 4 basic types of NoSQL databases:
1. Key-Value Store – It has a Big Hash Table of keys & values.
2. Document-based Store- It stores documents made up of tagged elements.
3. Column-based Store- Each storage block contains data from only one column.
4. Graph-based-A network database that uses edges and nodes to represent and store data.

Document in MongoDB

MongoDB stores data records as BSON documents. BSON is a binary representation of JSON documents, though it contains more data types than JSON. MongoDB documents are composed of field-and-value pairs and have the following structure: The value of a field can be any of the BSON data types, including other documents, arrays, and arrays of documents.

Relationship of MySQL terminology with MongoDB

The _id field represents a unique value in the MongoDB document.
Collection – This is a grouping of MongoDB documents.
Database – This is a container for collections like in RDMS wherein it is a container for tables.
Document - A record in a MongoDB collection is basically called a document.
Field - A name-value pair in a document.

When to use MongoDB?

  • When you need high availability of data with automatic, fast and instant data recovery.
  • If you have an unstable schema and you want to reduce your schema migration cost.
  • If you don’t have a Database Administrator.
  • If most of your services are cloud-based, MongoDB is best suitable for you, as its native scale-out architecture enabled by ‘sharding’ aligns well with horizontal scaling and agility offered by cloud computing.

MongoDB: Pros & Cons

  • Absence of a schema gives you more flexibility - you can adjust your stored data at any point and introduce new "fields"
  • Data is stored in the format your app needs it - this speeds up fetching the data
  • Expansion plans are flexible and can be easily achieved by adding more machines and RAM to the system.
  • The cons of MongoDB include higher data size over the period of time.
